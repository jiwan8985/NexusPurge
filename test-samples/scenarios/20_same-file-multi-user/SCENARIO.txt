시나리오 20 — 여러 운영자가 동시에 같은 파일 업로드/Purge
================================================================

목적: 두 명 이상의 운영자가 동시에 동일 S3 경로에 업로드하거나
      Purge를 실행했을 때 최종 상태가 예측 가능한지 확인

[배경]
게임사 운영팀에서 실제로 발생하는 상황:
- 운영자 A, B가 각자 NexusPurge를 열어서 같은 파일 수정 중
- A가 먼저 업로드 + Purge
- B가 수 초 후 다른 내용으로 업로드 + Purge
- 최종 CDN에 A의 내용이 남을지 B의 내용이 남을지 불확실

[시뮬레이션 (단일 PC에서 순차 실행)]

1단계: 공통 초기 상태 구성
   shared-config.txt (Version 1) 업로드

2단계: 운영자 A 작업
   shared-config-user-a.txt → shared-config.txt 로 교체 업로드
   [선택 Purge] 즉시 실행
   CDN 확인: "Last Editor: 운영자A"

3단계: 운영자 B 작업 (A 완료 직후 실행)
   shared-config-user-b.txt → shared-config.txt 로 교체 업로드
   [선택 Purge] 즉시 실행
   CDN 확인: "Last Editor: 운영자B"

4단계: 최종 상태 확인
   최종적으로 B 내용이 반영되어야 함 (Last Write Wins)
   A의 Purge가 B의 업로드보다 늦게 처리된 경우 엣지 케이스 발생 가능

[기대 결과]
- Last Write Wins: 마지막 업로드(B)의 내용이 최종 반영
- Purge 순서는 업로드 순서와 무관하게 처리
- 두 번의 Purge 모두 LogPanel에 기록

[실패 케이스]
- A의 Purge 후 B 업로드 전에 사용자가 캐시 접근 → A 내용 캐시
- B 업로드 완료 후 B의 Purge가 빨리 완료됨
- A의 Purge가 B 업로드 이후 처리되어 B 내용 캐시 제거
  → CDN이 오리진에서 B 내용을 다시 가져와 재캐시 (최종적으로 B 내용 반영)

[권장 운영 정책]
동일 파일 수정 시 팀 내 작업 순서 조율 필요
Lock 기능 없이는 Last Write Wins 정책에 의존
