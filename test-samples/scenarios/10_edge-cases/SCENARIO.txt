================================================================
시나리오 10 — 엣지 케이스 Purge
================================================================

목적: 일반적이지 않은 상황에서 Purge 동작의 안정성 검증

------------------------------------------------------------------------
케이스 A — 동일 파일명, 다른 경로 동시 Purge
------------------------------------------------------------------------
파일: same-name-different-prefix/assets/style.txt
      same-name-different-prefix/backup/style.txt

[순서]
1. 두 파일 모두 업로드
2. 두 파일 동시 선택 → [선택 Purge]
3. Purge 경로가 각각 정확한 full path로 전달되는지 확인:
   - .../assets/style.txt
   - .../backup/style.txt
   (파일명만 보내면 혼용될 수 있음)

[기대 결과]
두 경로 모두 독립적으로 캐시 무효화 ✓

------------------------------------------------------------------------
케이스 B — Purge 직후 즉시 재업로드
------------------------------------------------------------------------
파일: purge-then-reupload/cycle-file.txt

[순서]
1. cycle-file.txt 업로드 → CDN 캐시 생성
2. [선택 Purge] 실행
3. Purge 완료 신호 확인 전에 즉시 동일 파일 재업로드
4. CDN에 정상적으로 재캐시되는지 확인

[기대 결과]
Purge와 재업로드가 겹쳐도 최종 상태는 최신 파일이 캐시됨 ✓

------------------------------------------------------------------------
케이스 C — 빠른 연속 업데이트 (rapid-update)
------------------------------------------------------------------------
파일: rapid-update/rapid-file.txt

[순서]
1. rapid-file.txt 를 Version 1 내용으로 업로드
2. 즉시 Version 2 내용으로 교체 업로드 (Overwrite + 자동 Purge)
3. Purge 완료 전에 즉시 Version 3 내용으로 재업로드
4. CDN 최종 상태 확인

[기대 결과]
CDN에 최종 버전(Version 3)이 반영됨 ✓
Purge가 중복으로 발생해도 오류 없이 처리됨 ✓

------------------------------------------------------------------------
케이스 D — 빈 파일 업로드 & Purge
------------------------------------------------------------------------
파일: empty-file/empty.txt (0 bytes)

[순서]
1. empty.txt (0 bytes) 업로드
2. S3에 정상 등록 확인
3. CDN 접근 시 200 OK + 빈 응답 확인
4. [선택 Purge] 실행
5. Purge 정상 처리 확인 (크기 0이어도 에러 없어야 함)

[기대 결과]
빈 파일도 정상 업로드 및 Purge 처리 ✓
