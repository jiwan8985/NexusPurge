# NexusPurge Codex 개발 수정 프롬프트

현재 프로젝트는 NexusPurge이다. 이 프로젝트는 완전히 새로운 CDN 툴을 만드는 것이 아니라, 기존 개발사가 개발 및 운영하던 CDN 업로드/Purge 툴을 대체하기 위한 신규 구현 프로젝트이다. 기존에는 고객사와 기존 개발사가 함께 CDN 툴을 개발했고, 해당 개발사가 툴 개발과 운영까지 담당했다. 하지만 계약 만료로 인해 기존 툴을 더 이상 사용할 수 없게 되었고, 고객사는 새로운 업체를 통해 기존에 사용하던 툴과 동일한 기능을 가진 신규 툴을 요구하고 있다. 따라서 NexusPurge의 1차 목표는 신규 기능 확장이 아니라 기존 CDN 업로드/Purge 툴의 기능 재현이다. 문서와 코드 구조 모두 이 방향으로 수정한다.

확정 범위는 다음과 같다. 스토리지는 AWS S3만 지원한다. 문서나 코드에 Tencent COS, S3-compatible storage, 기타 Object Storage가 1차 범위처럼 표현되어 있다면 AWS S3 중심으로 정리한다. 단, 향후 확장을 막을 필요는 없다. 지원 CDN은 AWS CloudFront, Akamai, LG U+ CDN, 효성 CDN 4개이다. 현재 코드에 CloudFront와 Akamai만 반영되어 있다면 LG U+ CDN과 효성 CDN을 provider, type, UI, config, backend adapter 구조에 추가한다. LG U+ CDN과 효성 CDN의 실제 API 문서 및 인증 방식은 아직 고객사로부터 전달받지 못한 상태이므로 실제 Purge API 호출은 추측 구현하지 않는다. LG U+ CDN과 효성 CDN은 Stub Adapter로 추가하고, 명확한 NotImplemented 에러를 반환한다. 에러 메시지는 “LG U+ CDN purge API is not implemented yet. API specification is required.”, “Hyosung CDN purge API is not implemented yet. API specification is required.”로 한다.

고객사 기술 미팅에서는 인증 관련 질문은 제외한다. 다만 실제 개발 범위에는 외부 인증 모듈 연동 준비 구조를 포함한다. 로그인 및 사용자 인증 기능은 NexusPurge에서 자체 구현하지 않고, 다른 툴 개발업체가 제공하는 인증 모듈을 연동하는 구조이다. 이번 작업에서는 자체 로그인 화면, 자체 계정 DB, 자체 비밀번호 인증을 만들지 않는다. 대신 외부 인증 모듈과 연동 가능한 Adapter 또는 Interface 구조만 준비한다. 고객사 기술 미팅 질문에서는 로그/결과 처리 상세 항목을 제외한다. 다만 실제 개발 범위에는 기본 로그/결과 처리 구조를 포함한다. 업로드 결과 표시, Purge 결과 표시, 파일별 성공/실패 상태 표시, CDN별 Purge Request ID 또는 Task ID 저장 가능 구조, 실패한 업로드 재시도 구조, 실패한 Purge 재시도 구조, 작업 이력 로컬 저장 구조, JSON 로그 저장, 추후 CSV Export 확장 가능 구조를 포함한다.

이번 고객사 기술팀 미팅은 인증 미팅이 아니다. 인증은 별도 업체와 별도 미팅 예정이다. 또한 미팅에서는 로그/결과 처리 상세 질문도 제외한다. 이번 미팅의 목적은 기존 CDN 업로드/Purge 툴의 기능을 재현하기 위해 필요한 S3/CDN 기술 스펙 확인이다. 문서에는 고객사 확인 필요 사항으로 다음을 정리한다. 기존 툴의 필수 기능은 AWS S3 업로드 + CDN Purge 자동화가 맞는지, 지원 CDN은 CloudFront/Akamai/LG U+/효성 CDN 4개가 맞는지, 기존 툴에서 업로드/다운로드/삭제/폴더 생성/이름 변경/Presigned URL 생성/Purge 기능 중 실제 사용하던 기능이 어디까지인지, S3 Bucket은 단일 Bucket인지 환경별로 분리되어 있는지, S3 Prefix 규칙이 있는지, Prefix 예시는 dev/, stage/, prod/, assets/ 등을 기준으로 확인할 것, 동일 파일 판단은 MD5/ETag 비교 기준이면 되는지, 업로드 시 Content-Type과 Cache-Control 자동 설정이 필요한지, S3 Key와 CDN URL 매핑 규칙이 어떻게 되는지, 예시는 s3://bucket/prod/a.js 에서 https://cdn.example.com/prod/a.js 로 매핑되는 구조, CDN별로 도메인 또는 Prefix가 다를 수 있는지, 하나의 S3 파일이 여러 CDN 도메인에 동시에 연결될 수 있는지, Query String이 붙은 URL도 Purge 대상에 포함해야 하는지, Purge는 기존 파일 덮어쓰기 시에만 자동 실행하면 되는지, 신규 업로드 파일도 Purge 대상인지, 삭제 파일도 Purge 대상인지, 파일 단위 Purge만 필요한지, /assets/* 같은 폴더/와일드카드 Purge도 필요한지, Purge 실행 전 대상 URL 목록을 미리 보여줘야 하는지, Purge 요청 후 완료 상태까지 확인해야 하는지 또는 요청 성공까지만 확인하면 되는지, CloudFront는 Distribution ID + Object Path 기준이면 되는지, Akamai는 URL Purge 방식인지 CP Code Purge 방식인지, LG U+ CDN Purge API 문서와 인증 방식을 공유받을 수 있는지, 효성 CDN Purge API 문서와 인증 방식을 공유받을 수 있는지, CDN별 1회 Purge URL 개수 제한이나 Rate Limit이 있는지를 정리한다.

README.md, PROJECT_ANALYSIS.md, MEETING_SUMMARY.md, TODO.md 또는 TODO 관련 문서를 확인하고 필요한 내용을 수정한다. 문서에는 NexusPurge가 기존 개발사가 운영하던 CDN 업로드/Purge 툴을 대체하기 위한 신규 구현 프로젝트이며, 1차 목표는 신규 기능 개발이 아니라 기존 툴 기능 재현이고, 지원 스토리지는 AWS S3로 한정하며, 지원 CDN은 AWS CloudFront, Akamai, LG U+ CDN, 효성 CDN 4개이고, 로그인/인증 기능은 별도 툴 개발업체의 인증 모듈을 연동하는 구조로 진행하며 NexusPurge 자체 계정/비밀번호 인증은 구현하지 않고, 고객사 기술 미팅에서는 인증과 로그 상세 항목을 제외하고 S3/CDN 기술 스펙만 확인하되, 실제 개발 범위에는 외부 인증 모듈 연동 준비, 사용자 Role 기반 권한 제어 구조, 업로드/Purge 결과 표시, 작업 로그 로컬 저장, 실패 항목 재시도 구조를 포함한다는 내용을 반영한다. 우선 확인 및 구현 대상은 기존 툴 기능 범위 재현, AWS S3 파일 탐색/업로드/다운로드/삭제, 폴더 생성, 이름 변경, Presigned URL 생성 여부 확인, S3 Bucket/Prefix 구조 대응, S3 Key와 CDN URL 매핑, 파일 덮어쓰기 기준 CDN Purge, CloudFront/Akamai/LG U+ CDN/효성 CDN Purge Adapter 구조, 외부 인증 모듈 연동 구조, 작업 로그 및 결과 처리 구조이다.

src/types/index.ts 또는 관련 타입 파일을 찾아 수정한다. CdnProvider 타입에 cloudfront, akamai, lguplus, hyosung을 모두 포함한다. 기존 타입 구조가 다르면 기존 스타일을 유지하되 반드시 lguplus와 hyosung을 추가한다. S3 Profile 타입 또는 Profile 관련 타입에도 LG U+ CDN과 효성 CDN 설정을 저장할 수 있는 optional 필드를 추가한다. 필요한 필드는 cdnProvider, cdnDistributionId, cdnDomain, akamaiClientToken, akamaiClientSecret, akamaiAccessToken, akamaiHost, lguplusApiKey, lguplusApiSecret, lguplusEndpoint, hyosungApiKey, hyosungApiSecret, hyosungEndpoint, purgeOnNewUpload, defaultCacheControl, contentTypeOverride이다. 기존 필드를 삭제하지 말고 기존 CloudFront/Akamai 설정이 깨지지 않게 하며, 타입 이름이 다르면 기존 네이밍을 유지하고 타입 에러가 생기지 않도록 관련 사용처도 함께 수정한다. 외부 인증 모듈 연동 준비용 타입으로 AuthUser, AuthSession, AuthAdapter를 추가한다. AuthUser는 id, name, email, organization, roles를 가진다. AuthSession은 user, accessToken, refreshToken, expiresAt을 가진다. AuthAdapter는 login, logout, refreshToken, getCurrentSession 메서드 구조를 가진다. 작업 결과/로그용 타입으로 OperationType, OperationStatus, FileOperationResult, CdnPurgeResult, OperationLog를 추가한다. OperationType은 upload, download, delete, mkdir, rename, purge, sync를 포함한다. OperationStatus는 pending, running, success, failed, partial을 포함한다. FileOperationResult는 path, operation, status, message, error, startedAt, finishedAt을 가진다. CdnPurgeResult는 provider, urls, status, requestId, taskId, error, startedAt, finishedAt을 가진다. OperationLog는 id, profileId, operation, status, bucket, prefix, files, purgeResults, startedAt, finishedAt을 가진다.

src/components/modals/ProfileModal.tsx 또는 Profile 설정 UI 파일을 찾아 수정한다. CDN Provider 선택 목록에 AWS CloudFront, Akamai, LG U+ CDN, 효성 CDN 4개가 모두 표시되게 한다. LG U+ CDN 선택 시 API Key, API Secret, Endpoint, CDN Domain 입력 필드를 추가한다. 효성 CDN 선택 시 API Key, API Secret, Endpoint, CDN Domain 입력 필드를 추가한다. 기존 CloudFront/Akamai 입력 필드 동작은 깨지지 않아야 한다.

인증 기능은 자체 구현하지 않지만, 외부 인증 모듈을 연결할 수 있는 Adapter/Interface 구조를 추가한다. 가능한 파일 구조는 src/services/auth/auth-adapter.ts, src/services/auth/external-auth-adapter.ts, src/services/auth/auth-types.ts 이며, 기존 프로젝트 구조에 맞게 배치해도 된다. AuthAdapter interface를 정의하고 ExternalAuthAdapter Stub을 구현한다. 현재는 실제 API 호출하지 않고, 인증 모듈 문서 수령 후 구현 가능하도록 TODO를 명시한다. 로그인/로그아웃/토큰 갱신/현재 세션 조회 메서드 구조를 준비하고 사용자 Role 기반 기능 제한을 위한 타입을 준비한다. 자체 계정 DB를 만들지 않고, 자체 로그인 화면을 새로 만들지 않고, 실제 인증 API 호출은 구현하지 않고, 외부 인증 모듈 문서 수령 후 연결 가능한 구조만 준비한다.

작업 로그 및 결과 처리 구조를 추가한다. 가능한 파일 구조는 src/services/operation-log/operation-log-types.ts, src/services/operation-log/operation-log-store.ts 이며, 기존 프로젝트 구조에 맞게 배치해도 된다. 업로드 작업 결과를 파일별로 기록하고, 다운로드/삭제/폴더 생성/이름 변경 작업 결과를 기록하고, Purge 작업 결과를 CDN별로 기록하고, Request ID/Task ID 저장 가능 구조를 만들고, 실패 항목과 실패 사유를 저장하고, 실패 항목 재시도에 필요한 데이터를 보존하고, 로컬 JSON 저장 구조와 최근 작업 로그 조회 구조를 준비한다. CSV Export는 TODO로 남긴다. 기존에 유사한 로그/상태 구조가 있다면 중복 구현하지 말고 기존 구조를 확장한다.

src-tauri/src/utils/config.rs 또는 CDN credential을 관리하는 Rust 파일을 찾아 수정한다. CdnCredentials enum이 있다면 Lguplus와 Hyosung variant를 추가한다. Lguplus는 api_key, api_secret, endpoint, cdn_domain을 가진다. Hyosung도 api_key, api_secret, endpoint, cdn_domain을 가진다. get_cdn_credentials() 또는 provider 문자열을 기준으로 credentials를 가져오는 함수가 있다면 lguplus와 hyosung provider도 처리하도록 수정한다. 필수 값이 없으면 “LG U+ CDN credentials are incomplete”, “Hyosung CDN credentials are incomplete” 같은 명확한 에러 메시지를 반환한다. 기존 cloudfront, akamai 동작을 깨지 말고 기존 secret 저장 방식이 있으면 그 방식을 유지한다. 여기서 말하는 credential은 CDN API 호출용 설정이며 인증/로그인 모듈과 혼동하지 않는다.

src-tauri/src/adapters/cdn/lguplus.rs, src-tauri/src/adapters/cdn/hyosung.rs 파일을 추가한다. src-tauri/src/adapters/cdn/mod.rs에 pub mod lguplus; pub mod hyosung; 모듈을 등록한다. LG U+ CDN과 효성 CDN은 아직 API 문서가 없으므로 실제 Purge 호출을 구현하지 말고 명확하게 NotImplemented 에러를 반환하는 Stub으로 작성한다. LguplusCdnAdapter는 api_key, api_secret, endpoint를 가지고 new()와 purge_urls()를 제공한다. purge_urls()는 “LG U+ CDN purge API is not implemented yet. API specification is required.” 에러를 반환한다. HyosungCdnAdapter도 동일하게 api_key, api_secret, endpoint를 가지고 new()와 purge_urls()를 제공하며, purge_urls()는 “Hyosung CDN purge API is not implemented yet. API specification is required.” 에러를 반환한다. 프로젝트의 기존 에러 타입이 anyhow::Result, thiserror, custom error 등을 사용한다면 기존 스타일에 맞춘다.

src-tauri/src/adapters/cdn/mod.rs 또는 CDN Purge를 dispatch하는 파일을 찾아 수정한다. 현재 CloudFront/Akamai 분기가 있다면 LG U+ CDN과 효성 CDN도 분기 처리한다. Lguplus credential이면 build_cdn_urls로 cdn_domain과 paths를 URL 목록으로 변환하고 LguplusCdnAdapter::new(api_key, api_secret, endpoint)를 생성한 뒤 purge_urls를 호출한다. Hyosung credential이면 동일하게 HyosungCdnAdapter를 사용한다. 실제 프로젝트 함수 반환 타입에 맞춰 수정한다.

S3 Key를 CDN URL로 변환하는 로직이 여러 곳에 흩어져 있다면 공통 함수로 분리한다. 함수는 build_cdn_url(cdn_domain, object_path)와 build_cdn_urls(cdn_domain, paths) 형태로 만든다. build_cdn_url은 cdn_domain의 trailing slash를 제거하고 object_path의 leading slash를 제거한 뒤 “domain/path” 형태로 합친다. CloudFront는 Object Path 방식이므로 기존 방식이 있으면 유지한다. Akamai/LG U+/효성은 URL Purge 기반으로 사용할 수 있게 공통 URL 생성 함수를 사용한다. 고객사 확인 전까지 Query String 처리는 구현하지 말고 TODO로 남긴다. CDN별 Prefix 차이도 고객사 확인 후 반영해야 하므로 TODO로 남긴다.

Rust 쪽에도 작업 로그 저장 모듈을 추가하거나 기존 구조를 확장한다. 가능한 파일 구조는 src-tauri/src/services/operation_log.rs 이다. 작업 로그를 로컬 app data 디렉터리에 JSON으로 저장하고, 최근 작업 로그를 조회하고, 특정 작업 로그 상세를 조회하고, 실패 항목 재시도에 필요한 데이터를 보존하는 구조를 만든다. CSV Export는 TODO로 남긴다. 프로젝트에 이미 유사한 로그/상태 구조가 있다면 중복 구현하지 말고 기존 구조를 확장한다.

Purge 정책은 기본적으로 기존 파일 덮어쓰기 시 Purge 기준으로 유지한다. 기존에 purgeOnNewUpload 옵션이 있다면 유지한다. 기본 정책은 기존 파일 덮어쓰기는 Purge 대상, 신규 업로드는 기본적으로 Purge 제외, 삭제 파일은 고객사 확인 후 반영, 와일드카드 Purge는 CDN별 지원 여부 확인 후 반영이다. 삭제 파일 Purge, 와일드카드 Purge, Query String Purge, CDN별 Prefix 상이 케이스, Purge 완료 상태 Polling은 이번 작업에서 바로 구현하지 말고 TODO로 남긴다.

문서/TODO에 기능 우선순위를 반영한다. 1차 우선순위는 기존 툴 대체 목적 문서화, AWS S3 중심 구조 정리, CloudFront 기존 기능 유지, Akamai 기존 기능 유지, LG U+ CDN Provider 추가, 효성 CDN Provider 추가, LG U+ CDN Stub Adapter 추가, 효성 CDN Stub Adapter 추가, Profile UI에 4개 CDN 선택 가능하도록 수정, S3 Key → CDN URL 매핑 공통 함수 추가, 외부 인증 모듈 연동 Adapter 구조 추가, 작업 로그/결과 타입 및 저장 구조 추가이다. 고객사 확인 후 구현 항목은 Akamai URL Purge/CP Code Purge 방식 확정, LG U+ CDN Purge API 실제 구현, 효성 CDN Purge API 실제 구현, S3 Bucket/Prefix 규칙 반영, S3 Key ↔ CDN URL 매핑 세부 규칙 반영, 삭제 파일 Purge 정책 반영, 폴더/와일드카드 Purge 지원 여부 반영, Query String Purge 처리 여부 반영, Purge 완료 상태 Polling 여부 반영이다. 별도 인증 업체 미팅 후 구현 항목은 실제 인증 모듈 API 연동, 로그인/로그아웃 실제 구현, Access Token/Refresh Token 처리, 사용자 Role과 NexusPurge 기능 권한 매핑, 인증 토큰 저장/갱신/만료 처리이다. 후속 고도화 항목은 CSV Export, 상세 작업 리포트, 감사 로그, 관리자용 작업 이력 조회, 재시도 Dashboard이다.

절대 하지 말아야 할 것은 다음과 같다. Tencent COS 지원을 1차 범위처럼 문서화하지 않는다. S3-compatible storage를 1차 핵심 범위처럼 강조하지 않는다. 인증/로그인을 NexusPurge 자체 계정 DB 방식으로 구현하지 않는다. 자체 로그인 화면을 새로 만들지 않는다. LG U+ CDN/효성 CDN 실제 API 호출을 추측으로 구현하지 않는다. CloudFront 기존 구현을 깨지 않는다. Akamai 기존 구현을 삭제하지 않는다. 기존 타입/설정 필드를 임의로 제거하지 않는다. 고객사 확인이 필요한 항목을 확정된 것처럼 실제 구현하지 않는다. 미팅 질문 리스트에 인증/로그 상세 항목을 다시 포함하지 않는다. 단, 실제 개발 구조에는 인증 연동 준비와 로그/결과 처리를 포함한다.

작업 완료 후 가능한 범위에서 pnpm typecheck, pnpm lint, cargo check를 실행한다. 프로젝트에 해당 스크립트가 없으면 가능한 대체 명령을 사용한다. 실패 시 실패 이유를 요약하고, 수정 가능한 범위는 수정한다.
